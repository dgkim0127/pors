# Firebase Data Map

## 1. Purpose

This document describes how pors POS Firebase sales data can be used by Noblesse web for buyer analytics.

The POS app remains a tablet APK app.
The Noblesse website remains a B2B catalog and admin system.
The goal is data connection, not UI merging.

## 2. Firebase Project

Known Firebase project:
- `pors-piercing-pos`

Current repository findings:
- Firebase initialization file path: `src/standalone.js`
- Firebase runtime config file path: `public/app-config.js`
- Firestore usage file path: `src/standalone.js`
- Built/static copy with the same Firestore logic: `public/standalone.js`
- Firebase SDK loading: `index.html` uses Firebase compat scripts from `www.gstatic.com`
- Firebase code exists in this repository: yes
- Current `main` includes Firebase sync code: yes, based on the code inspected in this branch created from `origin/main`

The app uses the browser Firebase compat API:
- `window.firebase.initializeApp(config)`
- `window.firebase.firestore()`
- `db.collection(name)`
- document `set`, `get`, `delete`
- collection `get`
- collection/document `onSnapshot`
- `orderBy("createdAt", "desc")` for sales
- `orderBy("deletedAt", "desc")` for deletion logs

The legacy whole-state document is also configured:
- collection: `window.PIERCE_FIREBASE_COLLECTION || "piercing_pos"`
- document: `window.PIERCE_FIREBASE_DOC || "pors_state"`
- configured in `public/app-config.js` as `piercing_pos/pors_state`

## 4. Sales Document Shape

Sales are stored in the `sales` collection. The document id is `String(sale.id)`, and the document body is the sale object plus sync metadata.

Observed fields:
- `id`: generated by `id("sale")`
- `createdAt`: ISO string from `new Date().toISOString()`
- `customerId`: selected POS customer id, or `customer_manual` for manual customer entry
- `customerName`: selected or manual POS customer name
- `writerName`: POS writer/staff display name
- `discountRate`: customer discount rate copied at sale time
- `customerNote`: customer rule summary copied at sale time
- `customerPricingRules`: copied from customer
- `customerDiscountRules`: copied from customer
- `vatEnabled`: sale-time VAT flag
- `vatExempt`: sale-time VAT exemption flag
- `shippingFee`: numeric value, currently set to `0` by the sale draft builder
- `offshoreSettlement`: object or `null`
- `lines`: embedded sale lines array
- `totals`: totals object
- `cloudUpdatedAt`: numeric timestamp written during Firestore sync
- `dataVersion`: current app data version

Observed `totals` fields:
- `subtotal`
- `discount`
- `supply`
- `vat`
- `total`

Some receipt and settlement code also reads `totals.shippingFee`, but current sale creation stores `shippingFee` at sale root and does not add it to `totals`.

Fields from the expected list that differ:
- `saleId` is stored as `id`.
- `saleDate` is stored as `createdAt`.
- totals are nested under `totals`, not root-level `subtotal`, `discount`, `supply`, `vat`, `total`.
- sale lines are embedded as `lines[]`.
- `storeId` is not present on sale documents.
- `buyerId` is not present; `customerId` is present.

## 5. Sale Line Shape

Current structure is Option A: sale lines are embedded inside the `sales` document as `lines[]`.

No separate Firestore `saleLines` or `sale_items` collection is used by the current Firebase code.

Observed line fields:
- `id`: generated by `id("line")`
- `itemId`: POS item id, or generated manual item id for forced manual lines
- `categoryId`: POS category id
- `groupName`: POS subcategory/group display name
- `name`: sale-time line display name
- `price`: sale-time unit price after customer/item pricing rules
- `originalPrice`: original unit price when a direct line discount is applied, otherwise `null`
- `quantity`: sale quantity
- `discountable`: sale-time discount eligibility flag

Expected fields that differ:
- `lineId` is stored as `id`.
- `productId` is not present; POS uses `itemId`.
- `itemName`/`productName` is stored as `name`.
- `productCode` is not present.
- `unitPrice` is stored as `price`.
- `lineTotal` is not stored; calculate as `price * quantity`.

## 8. Mapping to Noblesse

Recommended mapping:

| pors POS | Noblesse analytics concept | Notes |
| --- | --- | --- |
| `customers` document | Noblesse buyer | Use `customerId`/customer `id` where available. |
| `items` document | Noblesse product or POS local product | Use `productCode` in the future when available. Current data does not store it. |
| `sales` document | POS sale record | Use `createdAt`, `customerId`, `customerName`, `totals`, and embedded `lines[]`. |
| `sales.lines[]` entry | POS sale item | Join to `items` by `itemId` when available. |
| `categories` document | product category | Join by line or item `categoryId`. |
| `subcategories` document | product subcategory/group | Join by `categoryId` plus group name/id where useful. |

Important mapping rules:
- `productCode` should be used to connect POS item to Noblesse `NB-*` product when available.
- Current Firebase data does not include `productCode`, so Noblesse should treat POS item matching as local/POS-only unless a future code field is added.
- `customerId` should be used to connect POS customer to Noblesse buyer when available.
- `customerName` is fallback only.
- Manual customer sales use `customer_manual`; those should be treated as unresolved buyer records unless matched later.
- Local-only POS items should be marked as local/custom in future analytics.

## 9. Analytics Fields Needed by Noblesse

| Metric | Current Firebase data support | Notes |
| --- | --- | --- |
| `purchaseCount` | Yes | Count `sales` grouped by `customerId`, with `customerName` fallback. |
| `totalPurchaseAmount` | Yes | Sum `sale.totals.total`. |
| `averagePurchaseAmount` | Yes | `totalPurchaseAmount / purchaseCount`. |
| `latestPurchaseDate` | Yes | Max `sale.createdAt`. |
| `repeatIntervalDays` | Yes | Sort each buyer's `createdAt` values and calculate intervals. |
| `totalQuantity` | Yes | Sum `sale.lines[].quantity`. |
| `topProducts` | Partial | Can group by `lines[].itemId` and `lines[].name`; Noblesse product matching needs `productCode`. |
| `reducedProducts` | Partial | Use `originalPrice > price`, `discountable`, and category/customer discount rules. Explicit reduction reason is not stored. |
| `categoryPreference` | Yes | Use line `categoryId`, with optional join to `categories`. |
| `monthOverMonthAmountChange` | Yes | Use `createdAt` month and `totals.total`. |
| `monthOverMonthQuantityChange` | Yes | Use `createdAt` month and line quantities. |
| `inactiveBuyerFlag` | Yes | Derive from latest sale date plus the chosen inactivity threshold. |

## 10. Missing Fields

Recommended additions for a future implementation:
- `productCode`: required to connect POS item lines to Noblesse `NB-*` products.
- stable Noblesse `buyerId`: useful if customer ids differ between systems.
- `storeId`: required if more than one POS store/device writes to the same Firebase project.
- `deviceId`: useful for APK/device audit and conflict analysis.
- `saleDate`: optional if `createdAt` remains stable; useful as a normalized business date.
- `categoryId` on sale line: already present.
- `lineTotal`: not required but useful for simpler analytics exports.
- `syncedAt`: useful for sync monitoring.
- `updatedAt`: useful for changed records; current sync metadata uses `cloudUpdatedAt`.
- `createdAt`/`updatedAt` on customers and items: useful for admin history.
- deletion log normalized fields: useful for analytics reconciliation if deleted sales should be excluded or audited.

## 11. Security Notes

- POS sales data must not be public.
- Noblesse web should access analytics with admin permission only.
- Store/customer isolation is required before multi-store analytics.
- Firebase rules must prevent public reads.
- Production keys must not be committed.
- APK/device data security has physical-access limits.
- Client-side Firebase config is not a secret by itself, but Firestore rules and authenticated access must protect sales/customer data.
- Read-only analytics access from Noblesse should not mutate POS sales, customers, items, or management collections.
